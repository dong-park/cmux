# A. 백엔드 프로젝트 템플릿

백엔드 프로젝트인 경우, **기술 기획서(spec)**와 **코드 수정 로드맵(roadmap)** 2개 파일을 작성합니다.

## 파일 1: 기술 기획서 (spec)

출력 형식은 다음 템플릿을 절대 벗어나지 않습니다:

### 1. Summary
- 핵심 목표와 해결하려는 문제를 2~3줄로 요약합니다.

### 2. Requirements
#### 2.1 Functional Requirements
- 기능별 요구사항을 간결한 bullet 형태로 정리합니다.
- 각 기능의 입력/출력(간단 요약)을 포함합니다.

#### 2.2 Non-Functional Requirements
- 성능, 안정성, 보안 등 필요한 비기능 요구사항을 포함합니다.

### 3. Workflow
- 시스템 기준의 전체 플로우를 단계별로 정리합니다.
- 성공/실패 흐름을 명확히 구분합니다.

### 4. API Spec
다음 형식을 사용합니다:
```
[API 명칭]
METHOD /path
Request:
{
  ...
}
Response:
{
  ...
}
Errors:
- 400: ...
- 500: ...
```

### 5. Data Model
- 엔티티 목록과 필드(타입/설명)를 표로 작성합니다.
- 엔티티 간 관계도 포함합니다.

### 6. Edge Cases & Validation Rules
- 입력값 검증, 예외 처리, 경계 상황을 bullet으로 정리합니다.

### 7. Test Scenarios
- 주요 기능별 테스트 케이스를 작성합니다.
- 정상/예외 케이스를 구분합니다.

### 8. Dev Notes
- 구현 시 주의 사항, 트랜잭션/캐싱 전략 등 실무적 메모를 포함합니다.

## 파일 2: 코드 수정 로드맵 (roadmap)

출력 형식은 다음 템플릿을 절대 벗어나지 않습니다:

### 1. 수정 파일 목록
| 순서 | 파일 | 수정 위치 | 변경 유형 |
|------|------|----------|----------|
| 1 | `파일경로` | 라인번호 | 추가/수정/삭제 |

### 2. 의존성 분석
- 수정이 영향을 미치는 다른 모듈/파일을 파악합니다.
- 연쇄 수정이 필요한 경우 명시합니다.

### 3. 상세 수정 내용
각 파일별로 다음 형식을 사용:
- **파일**: `파일경로`
- **위치**: 라인번호 또는 함수명
- **변경 내용**: diff 형식으로 Before/After 명시

```diff
- 삭제되는 코드
+ 추가되는 코드
```

### 4. 데이터베이스 마이그레이션
- 필요한 SQL 스크립트 작성
- 롤백 스크립트 포함 (가능한 경우)

### 5. 체크리스트
- [ ] 배포 전 확인 항목
- [ ] 테스트 항목
- [ ] 롤백 계획

1. 실무 개발자가 바로 구현 가능한 수준으로 구체적이고 간결하게 작성합니다.
2. UX 기획 요소(페르소나, 스토리 등)는 포함하지 않습니다.
3. 문장은 한국어로 작성하되 기술적, 명확한 톤을 유지합니다.
4. 불필요한 말은 제외하고 표/리스트 중심으로 구성합니다.
5. 코드베이스를 탐색하여 실제 수정될 파일과 라인번호를 정확히 파악합니다.
6. 출력은 반드시 위 템플릿에만 맞춰야 합니다.

# B. 프론트엔드 프로젝트 템플릿

프론트엔드 프로젝트인 경우, **프론트엔드 구현 가이드** 1개 파일을 작성합니다.

## 프론트엔드 구현 가이드 템플릿

출력 형식은 다음 템플릿을 절대 벗어나지 않습니다:

### 1. Implementation Summary
- 핵심 기능 요약 (2~3줄)
- 사용할 주요 라이브러리/기술
- 백엔드 API 의존성 여부
- 주요 UI/UX 특징

### 2. Task Breakdown
체크리스트 형식으로 작업 분해:
- [ ] [FE] 패키지 설치 (필요시)
- [ ] [FE] 컴포넌트 생성
- [ ] [FE] 페이지 수정
- [ ] [FE] State 관리
- [ ] [FE] API 연동
- [ ] [TEST] 기능 테스트 항목들

### 3. Frontend Changes

#### 3.1 컴포넌트 구조
각 컴포넌트별로:
- **파일 경로**: 정확한 파일 경로 명시
- **Props**: 인터페이스 정의
- **주요 로직**: 핵심 기능을 단계별로 설명 (번호 목록)
- **사용 라이브러리**: react-barcode, react-to-print 등

#### 3.2 페이지 수정
수정할 페이지별로:
- **파일 경로**: 정확한 파일 경로
- **위치**: 대략적인 라인 번호 (예: line 316-320 부근)
- **변경 내용**: 버튼 추가, State 추가, 함수 추가 등

#### 3.3 State 관리
- Recoil atoms/selectors 추가 (필요시)
- Component local state 추가
- 상태 데이터 타입 정의

#### 3.4 API 연동 (필요시)
- 사용할 API 엔드포인트
- Request/Response 타입
- 에러 처리 방안

#### 3.5 스타일링
- SCSS module 파일 경로
- 주요 레이아웃 구조
- 반응형/인쇄 모드 등 특수 스타일
- TailwindCSS 활용 방안

### 4. Component Skeleton (Code)
실제 구현 가능한 코드 스켈레톤 제공:
- TypeScript 인터페이스 정의
- 컴포넌트 기본 구조
- 주요 함수 로직
- 주석으로 구현 의도 설명

각 코드 블록에는:
```typescript
// 파일 경로 주석
// src/components/ComponentName/index.tsx

// 1. import 섹션
// 2. interface 정의
// 3. 컴포넌트 함수
// 4. 주요 로직 (번호 주석으로 단계 표시)
// 5. JSX 렌더링
```

### 5. Critical Files
수정/생성이 필요한 핵심 파일 목록:
- `package.json` - 의존성 추가 (필요시)
- `src/components/...` - 신규 컴포넌트
- `src/pages/...` - 페이지 수정
- `src/recoils/...` - 상태 관리 (필요시)
- `src/utils/...` - 유틸리티 (필요시)

### 6. Testing Checklist
실무 테스트 시나리오:
- [ ] 기본 기능 동작 확인
- [ ] 엣지 케이스 처리 (빈 데이터, 에러 상황 등)
- [ ] 브라우저 호환성 (Chrome, Safari, Edge)
- [ ] 반응형 확인 (필요시)
- [ ] 성능 확인 (대량 데이터 처리 등)
- [ ] 접근성 확인 (필요시)

## 프론트엔드 작성 규칙

1. **프론트엔드 중심**: UI 컴포넌트, State, 이벤트 핸들링에 집중합니다.
2. **실용성**: 복사-붙여넣기 가능한 코드 스켈레톤을 제공합니다.
3. **구체성**: "버튼 추가"가 아니라 "line 316 부근에 faPrint 아이콘 버튼 추가" 수준으로 작성합니다.
4. **타입 안정성**: TypeScript 인터페이스를 명확히 정의합니다.
5. **React 18 패턴**: 함수형 컴포넌트, hooks 중심으로 작성합니다.
6. **프로젝트 규칙 준수**:
   - Path aliases (@components, @utils 등) 사용
   - 기존 컴포넌트 패턴 (Modal, Button 등) 재사용
   - adminAPI 유틸리티 사용
   - TailwindCSS + SCSS module 조합
7. **한국어 작성**: 기술 용어 외에는 한국어로 작성합니다.
8. **출력 형식 엄수**: 반드시 위 템플릿 구조를 따릅니다.

## 프론트엔드 코드베이스 탐색 가이드

문서 작성 전 다음을 확인합니다:
- 기존 컴포넌트 패턴 파악 (Modal, Button, Form 컴포넌트 등)
- API 호출 방식 확인 (adminAPI 사용법)
- State 관리 패턴 확인 (Recoil atoms/selectors)
- 스타일링 규칙 확인 (TailwindCSS + SCSS)
- 실제 수정될 파일의 정확한 경로와 구조 파악
