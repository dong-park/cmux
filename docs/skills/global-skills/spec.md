# Spec - 기능 명세 작성

코드 작성 전에 "무엇을 만들 것인가"를 명확히 정의하는 스킬입니다.

## 사용법

```
/spec 사용자 프로필 캐싱       # 독립 호출
```

dev-flow에서 Phase 1로 자동 호출될 수도 있습니다.

## cmux 프로토콜

```bash
CMUX="$(command -v cmux-dev || command -v cmux)"

# 시작
"$CMUX" set-status phase "SPEC" --icon "📋" --color "#3B82F6"
"$CMUX" set-progress 0.0 --label "Spec: 시작"
"$CMUX" workspace-memo set "## Spec: [기능명]\nStarted: [날짜]\n---"

# 진행: 항목 정의 시마다 progress 갱신 (0.0 → 1.0)

# 완료
"$CMUX" workspace-memo append "### Spec 완료 ✅\n- 목표: ...\n- 비목표: ...\n- 성공 기준: ...\n- 경계 조건: ...\n---"
"$CMUX" set-progress 1.0 --label "Spec: 완료"
"$CMUX" notify --title "Spec 완료" --body "[기능명]"

# history 기록 (전역 축적)
"$CMUX" history add --type phase --phase spec --summary "SPEC 완료: [기능명]"
# 각 결정사항마다:
"$CMUX" history add --type decision --phase spec --summary "[결정 내용]" --tags [D1,키워드]
```

## 프로세스

### 1. 컨텍스트 수집

사용자의 요청을 분석하고, 필요 시 기존 코드베이스를 탐색하여 관련 컨텍스트를 파악합니다.

```bash
# 관련 코드 탐색
"$CMUX" set-progress 0.1 --label "Spec: 컨텍스트 수집"
```

### 2. 명세 항목 정의

사용자와 대화하며 다음 항목을 정의합니다:

| 항목 | 설명 | 기준 |
|------|------|------|
| **목표** | 이 기능이 해결하는 문제 | 3문장 이내. 넘으면 쪼개라 |
| **비목표** | 명시적으로 범위 밖인 것 | 최소 1개. "안 한다"고 적지 않으면 나중에 범위 확장 |
| **성공 기준** | 측정 가능한 완료 조건 | "빨라야 한다" ✗ → "p95 < 200ms" ✓ |
| **경계 조건** | 엣지 케이스, 제약사항 | 입력 범위, 동시성, 실패 모드 |
| **기술 결정** | 사용할 기술/패턴 (선택) | why 포함: "Redis — TTL 자동 만료 필요" |

```bash
"$CMUX" set-progress 0.5 --label "Spec: 항목 정의 중"
```

### 3. 명세 확정

모든 항목이 채워지면 사용자에게 확인을 요청합니다.

```markdown
## 기능: [기능명]

### 목표
- [구체적인 문제]

### 비목표
- [범위 밖 항목]

### 성공 기준
- [측정 가능한 조건]

### 경계 조건
- [엣지 케이스]

### 기술 결정 (선택)
- [기술/패턴 + 이유]
```

## Gate

- [ ] 목표가 3문장 이내로 명확한가?
- [ ] 비목표가 최소 1개 있는가?
- [ ] 성공 기준이 모두 측정 가능한가?
- [ ] 사용자가 동의했는가?

## 합리화 방지

| 합리화 | 반론 |
|--------|------|
| "간단한 기능이라 spec 필요 없어" | 간단할수록 범위가 몰래 늘어남. 3줄이라도 적어라 |
| "코드 보면 알 수 있어" | 코드는 what, spec은 why. 둘 다 필요 |
| "시간 없어서 바로 코딩해야 해" | spec 없이 만든 코드를 고치는 시간이 더 길다 |
| "요구사항이 아직 확정 안 됐어" | 확정 안 된 것 자체를 적어라. 불확실성도 명세다 |
| "이전에 비슷한 거 했으니까" | 비슷한 것과 같은 것은 다르다. 차이점을 명세하라 |

## Red Flags

- 목표가 4문장 이상 → 기능을 쪼개야 함
- 비목표가 0개 → 범위가 무한 확장될 위험
- 성공 기준에 "잘", "빠르게", "적절히" 같은 형용사 → 측정 불가
- 사용자가 "그냥 해봐"라고 하면 → 최소 3줄 spec이라도 작성 후 진행
